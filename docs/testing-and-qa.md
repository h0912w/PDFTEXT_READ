# Testing and QA Criteria

## QA 기본 원칙
- 모든 코드 수정 후 QA를 실행한다.
- 모든 문서 수정 후에도 QA를 실행한다.
- QA는 단순 명령 실행 확인이 아니라, 실제 샘플 PDF를 끝까지 처리하고 정답과 비교하는 회귀 테스트여야 한다.

## QA 입력
- 샘플 PDF 폴더
- 정답 데이터 폴더
- 변경된 코드 또는 문서

## QA 절차
1. 샘플 PDF 목록을 로드한다.
2. 각 샘플 PDF에 대해 실제 파이프라인을 실행한다.
3. 추출 결과를 생성한다.
4. 정답 데이터를 로드한다.
5. 결과와 정답을 자동 비교한다.
6. 불일치 항목을 식별한다.
7. 실패 원인을 분류한다.
8. 코드 수정 / 문서 수정 / 규칙 수정 필요 여부를 기록한다.
9. 수정 후 QA를 재실행한다.
10. 모든 샘플이 100% 일치할 때까지 반복한다.

## 통과 기준
- 등록된 모든 샘플 PDF가 PASS여야 한다.
- 디지털 PDF 샘플과 스캔본 PDF 샘플이 모두 통과해야 한다.
- skipped item count는 0이어야 한다.
- 정답 데이터와 100% 정확히 일치해야 한다.
- QA 리포트와 실패 원인 분류가 생성되어야 한다.

## 실패 기준
- 샘플 하나라도 불일치
- skipped item 1개 이상
- 정답 데이터 누락
- QA 리포트 생성 실패
- 샘플 실행 중 예외 발생

## 완료 기준
- 기능이 실행된다.
- 출력 계약을 유지한다.
- 핵심 예외 케이스를 깨지 않는다.
- QA가 100% 통과한다.
- QA 실패 상태의 변경은 완료로 간주하지 않는다.

## 권장 QA 자산 구조
```text
/qa
 ├── /samples
 ├── /answers
 ├── /fixtures
 └── /reports
```

## 리포트 필수 항목
- 샘플명
- 문서 유형
- PASS/FAIL
- 불일치 항목 목록
- skipped item 수
- 실패 원인 분류
- 수정 피드백
